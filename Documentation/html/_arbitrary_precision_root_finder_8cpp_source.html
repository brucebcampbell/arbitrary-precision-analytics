29 /*
30 These functions all perform iterative root finding using derivatives:
31 
32 newton_raphson_iterateperforms second order Newton-Raphson iteration,
33 halley_iterate andschroeder_iterate perform third order Halley and Schroeder iteration.
34 The functions all take the same parameters:
35 
36 Parameters of the root finding functions
37 
38 F f
39 Type F must be a callable function object that accepts one parameter and returns a boost::math::tuple:
40 
41 For the second order iterative methods (Newton Raphson) the boost::math::tuple should have two elements containing the evaluation of the function and its first derivative.
42 
43 For the third order methods (Halley and Schroeder) the boost::math::tuple should have three elements containing the evaluation of the function and its first and second derivatives.
44 
45 T guess
46 The initial starting value. A good guess is crucial to quick convergence!
47 
48 T min
49 The minimum possible value for the result, this is used as an initial lower bracket.
50 
51 T max
52 The maximum possible value for the result, this is used as an initial upper bracket.
53 
54 int digits
55 The desired number of binary digits.
56 
57 uintmax_t max_iter
58 An optional maximum number of iterations to perform.
59 
60 When using these functions you should note that:
61 
62 Default max_iter = (std::numeric_limits<boost::uintmax_t>::max)() is effectively 'iterate for ever'!.
63 They may be very sensitive to the initial guess, typically they converge very rapidly if the initial guess has
64 two or three decimal digits correct. However convergence can be no better than bisection, or in some rare cases,
65 even worse than bisection if the initial guess is a long way from the correct value and the derivatives are close to zero.
66 These functions include special cases to handle zero first (and second where appropriate) derivatives, and fall back to
67 bisection in this case. However, it is helpful if functor F is defined to return an arbitrarily small value of the correct
68 sign rather than zero.
69 If the derivative at the current best guess for the result is infinite (or very close to being infinite) then these functions
70 may terminate prematurely. A large first derivative leads to a very small next step, triggering the termination condition.
71 Derivative based iteration may not be appropriate in such cases.
72 If the function is 'Really Well Behaved' (monotonic and has only one root) the bracket bounds min and max may as well be set
73 to the widest limits like zero and numeric_limits<T>::max().
74 But if the function more complex and may have more than one root or a pole, the choice of bounds is protection against
75 jumping out to seek the 'wrong' root.
76 These functions fall back to bisection if the next computed step would take the next value out of bounds. The bounds are
77 updated after each step to ensure this leads to convergence. However, a good initial guess backed up by asymptotically-tight
78 bounds will improve performance no end - rather than relying on bisection.
79 The value of digits is crucial to good performance of these functions, if it is set too high then at best you will get one
80 extra (unnecessary) iteration, and at worst the last few steps will proceed by bisection. Remember that the returned value can
81 never be more accurate than f(x) can be evaluated, and that if f(x) suffers from cancellation errors as it tends to zero then
82 the computed steps will be effectively random. The value of digits should be set so that iteration terminates before this
83 point: remember that for second and third order methods the number of correct digits in the result is increasing quite
84 substantially with each iteration, digits should be set by experiment so that the final iteration just takes the next
85 value into the zone where f(x) becomes inaccurate.
86 To get the binary digits of accuracy, use policies::get_max_root_iterations<Policy>()).
87 If you need some diagnostic output to see what is going on, you can #define BOOST_MATH_INSTRUMENT before the
88 #include <boost/math/tools/roots.hpp>, and also ensure that display of all the possibly significant digits with
89 cout.precision(std::numeric_limits<double>::max_digits10): but be warned, this may produce copious output!
90 Finally: you may well be able to do better than these functions by hand-coding the heuristics used so that they
91 are tailored to a specific function. You may also be able to compute the ratio of derivatives used by these methods
92 more efficiently than computing the derivatives themselves. As ever, algebraic simplification can be a big win.
93 
94 Newton Raphson Method
95 Given an initial guess x0 the subsequent values are computed using:
96 x_{n+1} = x{n}- \frac{f(x)}{f'(x)}
97 
98 Out of bounds steps revert to bisection of the current bounds.
99 
100 Under ideal conditions, the number of correct digits doubles with each iteration.
101 
102 Halley's Method
103 Given an initial guess x0 the subsequent values are computed using:
104 x_{n+1} = x{n}- \frac{2f(x)*f'(x)}{2(f'(x))^2 - f(x)*f''(x)}
105 
106 Over-compensation by the second derivative (one which would proceed in the wrong direction) causes the method to revert to a Newton-Raphson step.
107 
108 Out of bounds steps revert to bisection of the current bounds.
109 
110 Under ideal conditions, the number of correct digits trebles with each iteration.
111 
112 Schroeder's Method
113 Given an initial guess x0 the subsequent values are computed using:
114 x_{n+1} = x{n}- \frac{f(x)}{f'(x)} - \frac{f''(x)(f(x))^2}{2(f'(x))^3}
115 
116 Over-compensation by the second derivative (one which would proceed in the wrong direction) causes the method to revert to a Newton-Raphson step. Likewise a Newton step is used whenever that Newton step would change the next value by more than 10%.
117 
118 Out of bounds steps revert to bisection of the current bounds.
119 
120 Under ideal conditions, the number of correct digits trebles with each iteration.
121
